1. PROJECT OVERVIEW
================================================================================

BluePay is a Flutter-based mobile payment application designed specifically for
OFFLINE rural payments. It allows users to send and receive money even without
internet connectivity, using a clever combination of:

  - Bluetooth + WiFi Direct (Google Nearby Connections) for device-to-device
    local transfers
  - GSM SMS as a relay channel to notify the banking backend when a transaction
    happens
  - USSD-style dialpad as a fallback input method (*#amount#phone#)
  - QR Code for receiver identity sharing
  - Connectivity Radar (ShareIt-style discovery) for fast nearby peer detection

The app targets Android devices and is built to work in areas with poor or
zero internet connectivity — a common challenge in rural India.

4. ANDROID NATIVE LAYER (Kotlin)
================================================================================

  The app has 3 native Kotlin files that bridge Android APIs to Flutter:

  4.1 MainActivity.kt
  ───────────────────
  - Extends FlutterActivity (Flutter's Android entry point)
  - Registers 3 platform channels:

    a) AUDIO CHANNEL: "com.example.bluepay/audio"
       - MethodChannel for playing the success sound
       - Method: "playSuccessSound"
       - Implementation: Android MediaPlayer + FlutterInjector to load
         assets/audio/success.mp3

    b) SYSTEM CHANNEL: "com.example.bluepay/system"
       - MethodChannel for reading Android system settings
       - Method: "isAirplaneModeOn"
       - Implementation: Settings.Global.AIRPLANE_MODE_ON query
       - Used by SmsQueueService to decide whether to queue or send SMS

    c) SMS EVENT CHANNEL: "com.example.bluepay/sms_events"
       - EventChannel for streaming incoming BPAY SMS events to Flutter
       - Stream handler: SmsEventStreamHandler (singleton object)

  4.2 SmsBalanceReceiver.kt
  ─────────────────────────
  - Android BroadcastReceiver registered for SMS_RECEIVED intent
  - Filters incoming SMS for messages starting with "BPAY "
  - Format: "BPAY S/R {txn_id} {amount} {newBalance}"
  - When a BPAY SMS is detected → calls SmsEventStreamHandler.emit()
  - Registered in AndroidManifest.xml with priority 1000

  4.3 SmsEventStreamHandler.kt
  ─────────────────────────────
  - Kotlin singleton object implementing EventChannel.StreamHandler
  - Bridges SmsBalanceReceiver → Flutter EventChannel
  - emit() is dispatched on the main thread via Handler(Looper.getMainLooper())
  - Flutter's SmsBalanceService listens to this stream

  │    → Tap user from list → requestConnection()                        │

  └──────────────────────────────────────────────────────────────────────┘

  RECEIVER FLOW:
  ┌──────────────────────────────────────────────────────────────────────┐
  │  HomeDashboard → [Receive Money] → ReceiveScreen                     │
  │    → Nearby Advertising starts (phone number as identity)            │
  │    → QR code displayed (JSON: {phone, name})                         │
  │    → Sender scans + connects → acceptConnection()                    │
  │    → BYTES payload received → _handleIncomingPayment()              │
  │    → AppState.receiveMoney() → SmsQueueService.enqueue(JSON)         │
  │    → SuccessAnimation → Home                                          │

  ┌──────────────────────────────────────────────────────────────────────┐

  │    → Auto-detected by RegExp → Confirm dialog                        │

8. OFFLINE HANDLING STRATEGY
================================================================================

  The app uses a layered offline strategy:

  Layer 1 — Peer-to-Peer (Bluetooth/WiFi Direct):
    - Money transferred device-to-device via Nearby Connections
    - No internet needed AT ALL for the actual transfer
    - Both parties see immediate balance change in their local AppState

  Layer 2 — SMS Queue (GSM Relay):
    - After every transaction, a JSON SMS is sent to the relay phone
    - If airplane mode is ON → SMS is persisted to SharedPreferences queue
    - When airplane mode turns OFF → queue auto-flushes
    - Safety net: 30-second periodic timer checks and flushes if possible
    - Airplane mode detected natively (Settings.Global.AIRPLANE_MODE_ON)
      NOT via connectivity_plus (which only tracks internet data, not GSM radio)

  Layer 3 — USSD Dialpad:
    - If Bluetooth/WiFi is unavailable, user can manually type USSD pattern
    - *#<amount>#<phonenumber># → triggers SMS-only payment
    - Backend processes and confirms via return SMS

9. IDEMPOTENCY & DUPLICATE PREVENTION
================================================================================

  Transaction IDs:
    - Format: "TXN{System.currentTimeMillis()}{random4hex}"
    - Generated by SENDER before sending
    - Same txnId used in: Nearby payload, SMS to backend, AppState record

  Duplicate Guards:
    - AppState.receiveMoney() checks if txnId already exists in transactions list
    - AppState.sendMoney() same check
    - If duplicate → logs warning, returns without double-crediting/debiting
